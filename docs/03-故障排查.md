# 03 · 故障排查

> 遇到问题时先看这里。按错误类型分类。

---

## 一、飞书 API 错误

### 1.1 `feishu API failed: 99991663`

**原因**：飞书免费版 10000 次/月 API 调用上限

**处理**：
1. 飞书开放平台后台看调用次数统计
2. 减少 archive 频率（用 7 天安全期，少跑 `--purge-immediately`）
3. 接近配额时升级付费版

### 1.2 `feishu API failed: 99991661`（token 失效）

**原因**：tenant_access_token 过期或 SDK 刷新失败

**处理**：
1. 检查 `.env` 的 `FEISHU_APP_ID` / `FEISHU_APP_SECRET` 是否正确
2. 检查应用是否被禁用
3. 重试

### 1.3 `field validation failed (99992402)`

**原因**：bitable 字段值格式不对（如单选字段传了不在选项里的值）

**处理**：
1. 看 sync 跑完后是否有 warnings
2. 跑 `python deploy.py fix-bitable` 修复单选字段选项
3. 在 bitable UI 检查具体记录的字段值

### 1.4 `feishu API failed: 1254030`（无权限）

**原因**：应用没有对应资源的权限

**处理**：
1. 飞书开放平台 → 应用 → 权限管理，确认权限齐全：
   - `wiki:wiki` / `docx:document` / `bitable:app` / `drive:drive` / `drive:media:upload`
2. 应用发布版本是否包含这些权限
3. 应用是否被加到对应资源（知识空间 / bitable）的协作者里

## 二、OSS 错误

### 2.1 `AccessDenied` 点击下载链接

**原因**：bucket ACL 是私有，但用了私有 sign_url 或没设 OSS_PUBLIC_BASE

**处理**：
1. 阿里云控制台 → bucket → 权限管理 → Bucket ACL → 改「公共读」
2. `.env` 加 `OSS_PUBLIC_BASE=https://your-bucket.oss-cn-beijing.aliyuncs.com`
3. 重跑 `archive-materials` 让新归档用纯直链
4. 已经归档的：在 bitable UI 改资料表「文件链接」字段，把 URL 里的签名部分删掉，只保留 `https://bucket/raw/...`

### 2.2 `FileUploadException: OSS 上传失败`

**原因**：AK/SK 错 / 网络问题 / bucket 不存在

**处理**：
1. 检查 `.env` 的 `OSS_ACCESS_KEY_ID` / `OSS_ACCESS_KEY_SECRET`
2. 检查 `OSS_BUCKET` 名称拼写
3. 检查 `OSS_ENDPOINT` 与 bucket 区域一致
4. 阿里云控制台 → RAM → 用户 → 确认 AK/SK 有 `AliyunOSSFullAccess` 权限

### 2.3 `FileUploadException: InvalidAccessKeyId`

**原因**：AK/SK 错或被禁用

**处理**：阿里云控制台 → RAM → 用户 → AccessKey 管理 → 启用 / 新建

## 三、OCR / LLM 错误

### 3.1 `LibreOffice 未安装，跳过 Office 文件 OCR`

**原因**：LibreOffice 没装

**处理**：
- Windows：https://www.libreoffice.org/dl/win-x86_64/ 下载安装
- macOS：`brew install libreoffice`
- Linux：`apt install libreoffice`

装完验证：`soffice --version`

### 3.2 `LibreOffice 转换超时（120s）`

**原因**：文件太大或 LibreOffice 卡了

**处理**：
1. 文件超大（> 50MB）：考虑拆分
2. LibreOffice 卡：重启 LibreOffice 进程（杀掉所有 soffice 进程）
3. 改代码 timeout 参数（services/ocr_service.py:51）

### 3.3 GLM-OCR 失败 / 返回空

**原因**：智谱 API 余额不足 / 网络 / 文件格式不支持

**处理**：
1. 智谱开放平台 → 看余额
2. 检查 `.env` 的 `GLM_API_KEY` 是否有效
3. PDF 是否加密 / 损坏（用 PDF 阅读器打开验证）
4. 图片格式（PNG/JPG）暂不支持 GLM-OCR layout_parsing，跳过即可

### 3.4 LLM 摘要失败 / 返回乱码

**原因**：DeepSeek 余额不足 / 网络 / prompt 超长

**处理**：
1. DeepSeek 后台看余额
2. 检查 `.env` 的 `LLM_API_KEY`
3. OCR 全文是否超长（> 50000 字符），需要先截断

### 3.5 `zipfile.BadZipFile: File is not a zip file`

**原因**：你的 `.xlsx` 文件实际不是真的 xlsx（可能是 GBK 编码的 TSV / CSV）

**处理**：
- 课程清单文件：代码会自动 fallback 到文本格式（GBK/UTF-8 sniff），无需处理
- 其他 xlsx 文件：用 Excel 重新「另存为 .xlsx」格式

## 四、文档生成错误

### 4.1 sync 跳过资料 + 警告

**症状**：`skipped_materials: N`，warnings 显示「课程 X 不在主数据表」

**原因**：学生填的课程名拼写错，或新课没录入

**处理**：
1. 拼写错：让学生重填，或管理员在 UI 改资料表「课程」字段
2. 新课：管理员在 bitable UI 课程主数据表加这门课，重跑 sync

### 4.2 docs 生成的文档打不开 / 找不到

**症状**：点链接提示无权限，或在飞书云文档找不到

**原因**：docx 建在应用 owner 的「我的空间」根目录

**处理**（当前已知问题，生产化清单 2.1h）：
1. 把应用 owner 改成你常用的飞书账号
2. 或在飞书 UI 把孤儿文档移动到共享目录
3. 长期方案：配置 `folder_token` 让 docs 建在指定目录

### 4.3 docs 重跑产生大量孤儿文档

**原因**：飞书 API 不支持「覆盖」，每次重跑都新建文档

**处理**：
1. 接受现状（孤儿不影响功能，只是占空间）
2. 定期去飞书云文档清理（应用 owner 的「我的空间」根目录）
3. 长期方案：配置 `folder_token` 让所有 docx 集中在一个目录，方便批量管理

### 4.4 `replace_year_overview 失败: invalid param`

**症状**：学年总论段落追加失败

**原因**：旧块定位逻辑（按 heading text）和实际不符

**处理**：暂留为已知问题。学年总论暂时手动编辑学年文档。

## 五、bitable 错误

### 5.1 `未找到 资料管理表`

**原因**：bitable 还没初始化

**处理**：跑 `python deploy.py init-bitable`

### 5.2 bitable UI 看不到记录

**原因**：应用是 owner，但当前用户没加为协作者

**处理**：
```bash
python deploy.py grant-bitable --type email your-email@example.com --perm full_access
```

或更简单：
```bash
python deploy.py open-bitable -e anyone_editable
```

后者凭链接即可编辑（demo 推荐，生产不推荐）。

### 5.3 单选字段没有选项

**症状**：bitable UI 单选字段下拉是空

**处理**：
```bash
python deploy.py fix-bitable
```

会自动从 `config/course_schema.py` 的 `SINGLE_SELECT_OPTIONS` 补全选项。

## 六、其它

### 6.1 PowerShell 中文乱码

**不影响功能**：仅控制台显示问题（GBK/UTF-8 冲突）

**缓解**：`chcp 65001` 切到 UTF-8，或用 Windows Terminal

### 6.2 structlog 日志重复

**症状**：每条日志打印两次（一次纯文本，一次带 kwargs）

**原因**：structlog 默认配置

**处理**：暂留（生产化清单 4.5）。功能不受影响。

### 6.3 路径含中文报错

**症状**：`FileNotFoundError` 或 `UnicodeDecodeError`，但路径看起来正确

**处理**：
1. Python 脚本开头加 `sys.stdout.reconfigure(encoding='utf-8')`
2. 文件操作始终指定 `encoding='utf-8', newline='\n'`
3. 用 `os.path.exists()` 验证路径，不依赖 print 输出（中文路径在 PowerShell 显示乱码）

## 七、还不行？

1. 看 `python deploy.py logs` 最近失败的操作
2. 看 `logs/app.log` 完整日志
3. 看 CLAUDE.md 项目规范 + 本 docs 目录其他文档
4. 提 issue 到项目仓库

报告 bug 时附上：
- 命令行：你跑的完整命令
- 错误：完整 traceback
- 期望：你期望发生什么
- 实际：实际发生了什么
- 环境：Python 版本、OS、是否装了 LibreOffice
