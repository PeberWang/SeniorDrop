# 01 · 管理员指南

> 目标读者：负责日常运维的同学（班委 / 学习委员 / 项目负责人）。
>
> 前置：已完成 [00-快速开始.md](00-快速开始.md) 跑通首次部署。

## 一、日常运维：学生提交了新资料后

学生通过飞书表单提交了资料 → 资料管理表多了一条「待审核」记录 → 你的标准处理流程：

### Step 0：审核（在飞书 UI）

1. 打开 bitable（链接在你 `.env` 的 `BITABLE_APP_TOKEN` 里，URL 格式：`https://feishu.cn/base/{app_token}`）
2. 进入「资料管理表」
3. 看新提交的记录，检查：
   - 课程名是否正确（拼写 / 是否真有这门课）
   - 资料类型是否合理
   - 推荐理由是否清晰
   - 附件是否能打开
4. 把「审核状态」从「待审核」改为「已通过」（不合适的改「已拒绝」）

### Step 1：归档（飞书附件 → OSS）

```bash
python deploy.py archive-materials --purge-immediately
```

参数说明：
- `--purge-immediately`：立即删飞书原件（OSS 归档成功 + 回填 URL 后）。生产建议不加这个参数，保留 7 天安全期，定期跑 `purge-archived` 清理

跑完检查：
- 资料管理表的「文件链接」字段有 OSS URL（公共读直链）
- 「归档时间」字段有值
- 加了 `--purge-immediately` 的话，「文件附件」字段清空

### Step 2：OCR + 摘要

```bash
python deploy.py ocr-materials
```

这条命令对每份已归档资料：
1. 从 OSS 下载源文件
2. 检测格式：
   - PDF → 直接 OCR
   - Word/Excel/PPT → LibreOffice 转 PDF → OCR
   - zip → 解压 → 递归处理每个文件
3. GLM-OCR 提取全文
4. DeepSeek 生成摘要
5. 摘要 md 存飞书 drive + OSS
6. 回填资料表的「资料摘要」字段

跑完检查：
- 资料管理表的「资料摘要」字段有内容
- 时间：每份 PDF 约 30 秒-2 分钟（GLM-OCR 较慢）

### Step 3：sync（聚合 + 门控 + 更新派生字段）

```bash
python deploy.py sync
```

这条命令：
1. 扫三张表，构造 in-memory CourseData 列表
2. 双层门控：资料表/心得表的课程不在主数据表 → 跳过 + 警告
3. 更新主数据表的「资料数量」+「最后更新」字段

跑完看日志：
- `skipped_materials`：被门控跳过的资料数（应该 = 0；非 0 说明有学生填了主数据表里没有的课）
- `warnings`：具体跳过原因，按提示处理

### Step 4：重跑 docs + link

```bash
python deploy.py docs
python deploy.py link
```

注意：docs 会创建新的课程文档（飞书 API 不支持覆盖），旧文档会变成孤儿。如需清理，去飞书云文档手动删。

跑完检查：
- 课程文档里有了新的「推荐资料」表 + 「资料串讲」段落（含可点击下载链接）
- 导航表的「学习指南」字段链接更新

## 二、学生反馈「没有我的课程」

飞书表单单选字段不支持「其他 + 自由填写」。学生下拉里只能选已录入的课。处理：

1. 学生通过班委 / 群反馈新课名
2. 你在 bitable UI 课程主数据表加一条记录：
   - 课程名称（必填）
   - 授课老师
   - 开课学期（如「大二下」，下拉里选）
   - 课程类型（专业必修课 / 非专业必修课）
   - 考试形式（闭卷 / 开卷 / 论论 / 其他）
3. **关键**：飞书表单的字段选项会自动从主数据表读取（实际上是从字段配置读取，但 sync 会把主数据表的所有课程名写回资料表「课程」字段的可选值，需在 sync 跑完后看下资料表「课程」字段是否有这个新选项）
4. 学生重填表单 → 这次下拉里有了 → 提交
5. 走 Step 0-4 流程

如果学生已经提交了（但下拉里选不到，那他怎么提交？）—— 那他可能填了一个错误的课程名，sync 会跳过 + 警告，你按警告去主数据表加这门课，重跑 sync + docs + link 即可。

## 三、批量录入历史资料（首次填充用）

如果你有一批历史资料要批量录入（不是通过学生表单），用 `seed-materials`：

```bash
python deploy.py seed-materials "本地资料文件夹路径" 课程名 \
    --contributor "管理员" \
    --grade "22级" \
    --type "笔记" \
    --reason "推荐理由（一对多：本批次所有文件共享同一段理由）"
```

示例：

```bash
python deploy.py seed-materials "data/courses/大二上/概率论与数理统计（刘会刚老师）" 概率论与数理统计 \
    --grade 22级 \
    --contributor 测试管理员 \
    --reason "初始录入测试资料"
```

这条命令：
1. 递归扫描文件夹下所有文件（PDF / Word / Excel / PPT / zip 全部接受）
2. 每个文件上传到飞书 drive，拿到 file_token
3. 在资料管理表建一条记录，附件字段填 file_token
4. 所有文件共享同一段推荐理由（一对多）

跑完后，资料管理表有 N 条新记录（N = 文件数），审核状态默认「已通过」（因为是管理员录入）。然后跑 archive → ocr → sync → docs → link 完成全链路。

## 五、重置 / 重建（危险操作）

### 5.1 清空 bitable（保留表结构）

```bash
python deploy.py reset-bitable
```

会要求输入 `--yes` 确认。清空后：
- 三张表（课程主数据 / 资料管理 / 心得管理）的所有记录删除
- 表结构 + 字段定义保留
- OSS 上的归档资料**不删**（独立操作）

清空后必须重新 `seed-course` 录课程。

### 5.2 完全重建（包括知识库）

目前没有「一键重建知识库」命令，需要手动：
1. 飞书 UI 删除知识空间
2. 飞书 UI 删除应用 owner 的「我的空间」根目录下的所有 docx 孤儿
3. `reset-bitable`
4. 重新跑 init → seed → wiki → docs → link

## 六、权限管理

### 6.1 加协作者（精细权限）

```bash
python deploy.py grant-bitable --type email member@example.com --perm full_access
```

参数：
- `--type`：email / openid / userid / departmentid
- `--perm`：view（只读）/ edit（可改记录）/ full_access（可改字段定义）

### 6.2 凭链接访问（demo 模式）

```bash
python deploy.py open-bitable -e anyone_editable
```

参数：
- `closed`：不开放
- `tenant_readable`：组织内可读
- `tenant_editable`：组织内可编辑
- `anyone_readable`：凭链接任何人可读
- `anyone_editable`：凭链接任何人可编辑（demo 推荐）

生产环境请用 `grant-bitable` 精细化管理，不要用 `anyone_editable`。

## 七、常见问题处理

### 7.1 sync 跳过了资料

**症状**：`python deploy.py sync` 输出 `skipped_materials: 2`，warnings 显示「课程 X 不在主数据表」

**原因**：学生表单填的课程名拼写错，或者真是一门没录入的新课

**处理**：
1. 拼写错：让学生重填，或管理员在 UI 改资料表的「课程」字段
2. 新课：管理员在课程主数据表加这条记录，重跑 sync

### 7.2 docs 生成的文档打不开

**症状**：点击「学习指南」链接，提示无权限

**原因**：docx 建在应用 owner 的「我的空间」根目录，用户没权限

**处理**：当前是已知问题（生产化清单 2.1h）。临时方案：把文档移动到共享目录，或把应用 owner 改成你常用的账号

### 7.3 OCR 失败 / 摘要空

**症状**：`ocr-materials` 跑完后资料表的「资料摘要」字段还是空

**排查**：
1. 检查 `.env` 的 `GLM_API_KEY` 是否有效
2. 跑 `python deploy.py logs` 看 ocr_materials_pipeline 的错误
3. 智谱余额是否充足
4. LibreOffice 是否装了（Office 文件 OCR 必需）：`soffice --version`

### 7.4 飞书 API 配额超限

**症状**：`feishu API failed: 99991663` 或类似

**原因**：飞书免费版 10000 次/月 API 调用上限

**处理**：
1. 看飞书开放平台后台的调用次数统计
2. 减少 archive 频率（多用 7 天安全期，少跑 --purge-immediately）
3. 接近配额时升级付费版

更多问题详见 [03-故障排查.md](03-故障排查.md)。
